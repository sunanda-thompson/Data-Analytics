SQL Cheat Sheet — PostgreSQL vs MySQL

Side-by-side syntax reference for both databases. Identical syntax is shown once; differences are called out clearly.

Data Types
Column type definitions when creating tables
ConceptPostgreSQLMySQLNotes
Auto-increment ID BIGSERIAL PRIMARY KEY BIGINT AUTO_INCREMENT PRIMARY KEY Diff PG uses BIGSERIAL as shorthand. MySQL requires AUTO_INCREMENT explicitly.
Text (variable) VARCHAR(255) VARCHAR(255) Same Max 255 chars. Use for names, emails, labels.
Fixed-length text CHAR(2) CHAR(2) Same Good for state abbreviations (always exactly 2 chars).
Whole number INTEGER / BIGINT INTEGER / BIGINT Same Use BIGINT for IDs (future-proof). INTEGER for known-small values like month numbers.
Decimal number NUMERIC(14,2) DECIMAL(14,2) Diff Both work similarly. 14 = total digits, 2 = digits after decimal. Use for money.
Date only DATE DATE Same Format: YYYY-MM-DD
Date + Time TIMESTAMP DATETIME Diff PG also has TIMESTAMPTZ (with time zone). MySQL uses DATETIME.
True / False BOOLEAN TINYINT(1) Diff PG has a native BOOLEAN. MySQL uses 0/1 stored as TINYINT.

Table Operations (DDL)
Creating, modifying and removing tables
ConceptPostgreSQLMySQLNotes
Create a table CREATE TABLE employees ( id BIGINT PRIMARY KEY, name VARCHAR(255) ); CREATE TABLE employees ( id BIGINT PRIMARY KEY, name VARCHAR(255) ); Same
Drop table (safe) DROP TABLE IF EXISTS employees; DROP TABLE IF EXISTS employees; Same IF EXISTS prevents an error if the table doesn't exist.
Add a column ALTER TABLE employees ADD age INTEGER; ALTER TABLE employees ADD age INTEGER; Same
Rename a column ALTER TABLE employees RENAME age TO years_old; ALTER TABLE employees RENAME COLUMN age TO years_old; Diff MySQL 8.0+ supports RENAME COLUMN. Older versions use CHANGE.
Temp table CREATE TEMP TABLE temp_data AS SELECT * FROM sales; CREATE TEMPORARY TABLE temp_data SELECT * FROM sales; Diff PG: TEMP. MySQL: TEMPORARY. Both cleared on disconnect.

Insert / Update / Delete (DML)
Adding, modifying and removing rows
ConceptPostgreSQLMySQLNotes
Insert a row INSERT INTO employees VALUES (1, 'Bob', 40); INSERT INTO employees VALUES (1, 'Bob', 40); Same
Update a row UPDATE employees SET age = 41 WHERE name = 'Bob'; UPDATE employees SET age = 41 WHERE name = 'Bob'; Same ⚠️ Always include WHERE — no WHERE = every row gets updated.
Delete a row DELETE FROM employees WHERE name = 'Bob'; DELETE FROM employees WHERE name = 'Bob'; Same ⚠️ Same warning — no WHERE = all rows deleted.
Insert from SELECT INSERT INTO archive SELECT * FROM employees WHERE age > 60; INSERT INTO archive SELECT * FROM employees WHERE age > 60; Same Copy rows from one table into another.

SELECT & Filtering
Querying and filtering rows
ConceptPostgreSQLMySQLNotes
Select all columns SELECT * FROM employees; SELECT * FROM employees; Same
Select with alias SELECT e.name, e.age FROM employees e; SELECT e.name, e.age FROM employees e; Same e is a short alias for the table. Best practice when joining tables.
WHERE + AND / OR WHERE age > 30 AND dept = 'Sales'; WHERE age > 30 AND dept = 'Sales'; Same
BETWEEN WHERE age BETWEEN 30 AND 50; WHERE age BETWEEN 30 AND 50; Same Both endpoints are inclusive.
IN WHERE dept IN ('Sales', 'HR'); WHERE dept IN ('Sales', 'HR'); Same Cleaner than multiple ORs. Keep list under ~20 values.
LIKE (pattern) WHERE name LIKE 'A%'; -- case-sensitive WHERE name ILIKE 'a%'; -- case-insensitive WHERE name LIKE 'A%'; -- case-insensitive by default -- No ILIKE in MySQL Diff PG LIKE is case-sensitive; use ILIKE for case-insensitive. MySQL LIKE is case-insensitive by default. % = any chars, _ = one char.
NULL check WHERE address IS NULL; WHERE address IS NOT NULL; WHERE address IS NULL; WHERE address IS NOT NULL; Same Never use = NULL — it doesn't work in either database.
DISTINCT SELECT DISTINCT dept FROM employees; SELECT DISTINCT dept FROM employees; Same Removes duplicate values from the result.
LIMIT rows SELECT * FROM employees LIMIT 10; SELECT * FROM employees LIMIT 10; -- or: LIMIT 2, 1 (skip 2, return 1) Diff MySQL supports LIMIT offset, count syntax. PG uses LIMIT x OFFSET y.

Aggregate Functions, GROUP BY & HAVING
Summarising and grouping data
ConceptPostgreSQLMySQLNotes
Count / Sum / Avg SELECT COUNT(*), SUM(salary), AVG(salary), MIN(salary), MAX(salary) FROM employees; SELECT COUNT(*), SUM(salary), AVG(salary), MIN(salary), MAX(salary) FROM employees; Same COUNT(*) counts all rows. COUNT(col) skips NULLs.
GROUP BY SELECT dept, COUNT(*) AS total FROM employees GROUP BY dept ORDER BY total DESC; SELECT dept, COUNT(*) AS total FROM employees GROUP BY dept ORDER BY total DESC; Same Every non-aggregate column in SELECT must also be in GROUP BY.
HAVING SELECT dept, AVG(salary) AS avg_sal FROM employees GROUP BY dept HAVING AVG(salary) > 60000; SELECT dept, AVG(salary) AS avg_sal FROM employees GROUP BY dept HAVING avg_sal > 60000; Diff MySQL allows using the column alias in HAVING. PG requires the full expression again.

JOINs
Combining data from multiple tables
ConceptPostgreSQLMySQLNotes
INNER JOIN SELECT * FROM customers c JOIN orders o ON c.id = o.customer_id; SELECT * FROM customers c INNER JOIN orders o ON c.id = o.customer_id; Same Returns only rows with a match in BOTH tables. JOIN and INNER JOIN are identical.
LEFT JOIN SELECT * FROM customers c LEFT JOIN orders o ON c.id = o.customer_id; SELECT * FROM customers c LEFT JOIN orders o ON c.id = o.customer_id; Same All rows from left table. NULL where no match on right. Most commonly used JOIN.
RIGHT JOIN ... RIGHT JOIN orders o ON c.id = o.customer_id; ... RIGHT JOIN orders o ON c.id = o.customer_id; Same All rows from right table. Rarely used — swap table order and use LEFT JOIN instead.
FULL OUTER JOIN ... FULL OUTER JOIN orders o ON c.id = o.customer_id; -- Not natively supported. -- Emulate with UNION: SELECT * FROM a LEFT JOIN b ON ... UNION SELECT * FROM a RIGHT JOIN b ON ...; Diff PG supports FULL OUTER JOIN natively. MySQL does not — emulate with a UNION of LEFT + RIGHT joins.
Self JOIN SELECT e.name, mgr.name AS manager FROM employees e JOIN employees mgr ON e.manager_id = mgr.id; SELECT e.name, mgr.name AS manager FROM employees e JOIN employees mgr ON e.manager_id = mgr.id; Same Table joined to itself using two different aliases. Common for manager/employee relationships.

CASE, UNION & Subqueries
Conditional logic, combining results, nested queries
ConceptPostgreSQLMySQLNotes
CASE statement SELECT name, CASE WHEN salary < 50000 THEN 'low' WHEN salary < 100000 THEN 'mid' ELSE 'high' END AS pay_band FROM employees; SELECT name, CASE WHEN salary < 50000 THEN 'low' WHEN salary < 100000 THEN 'mid' ELSE 'high' END AS pay_band FROM employees; Same SQL's if/else. Evaluates top to bottom — first TRUE condition wins.
UNION / UNION ALL SELECT name FROM employees UNION SELECT name FROM contractors; -- UNION ALL to keep duplicates SELECT name FROM employees UNION SELECT name FROM contractors; -- UNION ALL to keep duplicates Same Stacks two result sets. Both queries must have same number of columns.
Subquery in WHERE SELECT * FROM employees WHERE dept_id IN ( SELECT id FROM departments WHERE name = 'Sales' ); SELECT * FROM employees WHERE dept_id IN ( SELECT id FROM departments WHERE name = 'Sales' ); Same Inner query runs first. Its result is used by the outer query.
CTE (WITH clause) WITH high_earners AS ( SELECT * FROM employees WHERE salary > 80000 ) SELECT * FROM high_earners; WITH high_earners AS ( SELECT * FROM employees WHERE salary > 80000 ) SELECT * FROM high_earners; Same Cleaner alternative to nested subqueries. MySQL support added in v8.0.

Window Functions
Calculations across related rows without collapsing them
ConceptPostgreSQLMySQLNotes
Aggregate OVER partition SELECT name, dept, salary, AVG(salary) OVER (PARTITION BY dept) AS dept_avg FROM employees; SELECT name, dept, salary, AVG(salary) OVER (PARTITION BY dept) AS dept_avg FROM employees; Same Adds the dept average to every row — unlike GROUP BY which collapses to one row per dept.
ROW_NUMBER ROW_NUMBER() OVER (PARTITION BY dept ORDER BY salary DESC) AS row_num ROW_NUMBER() OVER (PARTITION BY dept ORDER BY salary DESC) AS row_num Same Unique sequential number per group. No ties — always 1, 2, 3, 4...
RANK vs DENSE_RANK RANK() OVER(...) -- 1,1,3,4 DENSE_RANK() OVER(...) -- 1,1,2,3 RANK() OVER(...) -- 1,1,3,4 DENSE_RANK() OVER(...) -- 1,1,2,3 Same RANK skips numbers after a tie. DENSE_RANK never skips.
Running total SUM(sales) OVER (ORDER BY sale_date) AS running_total SUM(sales) OVER (ORDER BY sale_date) AS running_total Same Cumulative sum up to the current row. Add ORDER BY inside OVER() to make it run.
LAG / LEAD LAG(salary) OVER(ORDER BY id) -- prev row value (NULL at top) LEAD(salary) OVER(ORDER BY id) -- next row value (NULL at bottom) LAG(salary) OVER(ORDER BY id) -- prev row value (NULL at top) LEAD(salary) OVER(ORDER BY id) -- next row value (NULL at bottom) Same
NTILE (buckets) NTILE(4) OVER (ORDER BY salary) AS quartile NTILE(100) OVER (ORDER BY salary) AS percentile NTILE(4) OVER (ORDER BY salary) AS quartile NTILE(100) OVER (ORDER BY salary) AS percentile Same Divides rows into N equal buckets. NTILE(4) = quartiles.

String Functions
Cleaning, formatting and extracting text
ConceptPostgreSQLMySQLNotes
Upper / Lower UPPER(name), LOWER(name) UPPER(name), LOWER(name) Same Standardise text before comparing. Use lower(x) = 'value'.
Trim whitespace TRIM(name) -- both sides LTRIM(name) -- left only RTRIM(name) -- right only TRIM(name) -- both sides LTRIM(name) -- left only RTRIM(name) -- right only Same Removes invisible spaces. Invisible whitespace breaks WHERE clause matches.
String length LENGTH(name) -- or CHAR_LENGTH(name) LENGTH(name) -- bytes CHAR_LENGTH(name) -- characters Diff In MySQL, LENGTH returns byte count (differs for multi-byte chars). Use CHAR_LENGTH to count characters.
Concatenate CONCAT(first, ' ', last) -- or: first || ' ' || last CONCAT(first, ' ', last) -- || operator not supported Diff PG supports || pipe operator for concatenation. MySQL uses only CONCAT().
Substring SUBSTRING(name, 1, 3) -- chars 1 through 3 SUBSTRING(name, 1, 3) -- same; also LEFT(name, 3) Same Both 1-indexed. MySQL also has LEFT() and RIGHT() shortcuts.
Replace REPLACE(name, 'a', '@') REPLACE(name, 'a', '@') Same Replaces all occurrences of the second argument with the third.
Handle NULLs COALESCE(address, 'N/A') COALESCE(address, 'N/A') -- or IFNULL(address, 'N/A') Diff COALESCE works in both. MySQL also has IFNULL() as a shorthand for the two-argument case.
Split a string SPLIT_PART('2023M01', 'M', 1) -- returns '2023' SUBSTRING_INDEX('2023M01','M',1) -- returns '2023' Diff Different function names. PG: SPLIT_PART(str, delim, position). MySQL: SUBSTRING_INDEX(str, delim, count).

Date Functions
Working with dates, times and intervals
ConceptPostgreSQLMySQLNotes
Current date / time CURRENT_DATE -- date only CURRENT_TIMESTAMP -- date + time CURDATE() -- or CURRENT_DATE NOW() -- or CURRENT_TIMESTAMP Diff MySQL commonly uses NOW() and CURDATE(). PG uses ANSI standard names.
Date arithmetic order_date + INTERVAL '7 days' order_date - INTERVAL '1 month' DATE_ADD(order_date, INTERVAL 7 DAY) DATE_SUB(order_date, INTERVAL 1 MONTH) Diff PG adds/subtracts with +/- and quoted INTERVAL strings. MySQL uses DATE_ADD() and DATE_SUB().
Days between dates CURRENT_DATE - order_date -- returns integer (days) DATEDIFF(CURDATE(), order_date) -- returns integer (days) Diff PG subtracts dates directly. MySQL uses DATEDIFF().
Extract part of date DATE_PART('month', order_date) DATE_PART('year', order_date) MONTH(order_date) YEAR(order_date) DAY(order_date) Diff PG uses DATE_PART with a string. MySQL has dedicated functions for each part.
Truncate to month DATE_TRUNC('month', order_date) -- returns first day of month DATE_FORMAT(order_date, '%Y-%m-01') -- format as first of month Diff PG has DATE_TRUNC for clean period rounding. MySQL uses DATE_FORMAT with a pattern string.
Cast to date type CAST(order_ts AS DATE) -- or: order_ts::DATE (PG shorthand) CAST(order_ts AS DATE) -- or: DATE(order_ts) Same CAST works in both. PG also supports :: shorthand. MySQL also accepts DATE().

Comments & Other Syntax Differences
Small but important syntax variations
ConceptPostgreSQLMySQLNotes
Single-line comment -- This is a comment -- This is a comment # This is also a comment Diff MySQL supports # as an additional comment style. PG does not.
Multi-line comment /* This spans multiple lines */ /* This spans multiple lines */ Same
String quotes 'single quotes only' -- double quotes = identifiers 'single quotes' OR "double quotes" -- both work for strings Diff PG: single quotes for strings, double quotes for column/table names. MySQL allows both for strings.
Stored procedure delimiter -- No DELIMITER needed CREATE PROCEDURE my_proc() LANGUAGE SQL AS $$ SELECT 1; $$; DELIMITER $$ CREATE PROCEDURE my_proc() BEGIN SELECT 1; END $$ DELIMITER ; Diff MySQL requires changing the DELIMITER to $$ so semicolons inside the procedure don't end it early. PG uses $$ ... $$ dollar-quoting instead.
Case sensitivity -- Keywords: case-insensitive -- Identifiers: lower by default -- "MyTable" ≠ mytable -- Keywords: case-insensitive -- Table names: depends on OS -- Column names: always insensitive Diff PG folds unquoted identifiers to lowercase. MySQL table name case depends on the operating system. Both treat SQL keywords case-insensitively.
